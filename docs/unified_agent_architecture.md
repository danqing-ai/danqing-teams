# 统一 Agent 架构方案：一切皆工具，模型驱动一切

> 一种无需区分 Code/Work 模式的 AI Agent 系统设计

---

## 一、核心架构哲学

### 1.1 一切皆工具（Everything is a Tool）

| 传统概念 | 本架构中的统一抽象 |
|---------|------------------|
| Sub-Agent | `sub_agent` Tool：接收参数，返回结果，可被委派 |
| 用户交互 | `ask_user` Tool：带选项、推荐、字段，返回用户反馈 |
| 技能/能力 | `skill` Tool：封装特定领域能力，可被动态调用 |
| 知识检索 | `knowledge` Tool：RAG/向量检索，返回上下文 |
| 记忆访问 | `memory` Tool：读写 episodic/semantic 记忆 |
| 文件操作 | `file` Tool：读/写/执行，权限由参数控制 |
| 外部 API | `api` Tool：统一封装，模型按需调用 |

**关键设计**：所有 Tool 具有统一接口签名，模型通过 Tool Call 驱动整个系统循环。

### 1.2 模型驱动一切（LLM-Centric Control）

```
用户输入
    ↓
[LLM 解析意图] → 规划 Tool Call DAG
    ↓
逐 Tool 执行（Agent Loop）
    ↓
需要澄清？→ ask_user Tool（上下文决定阻塞/非阻塞策略）
    ↓
需要委派？→ sub_agent Tool（参数决定资源配额和权限）
    ↓
完成 → 交付结果
```

**核心原则**：LLM 是唯一的决策中心，控制流由模型生成，而非开发者预设。

### 1.3 Tool Call 日志化（Persistent Execution Trace）

- 每一步 Tool Call 的输入、输出、耗时、决策理由完全持久化
- 支持异常恢复：任意步骤失败可从断点重试
- 支持可回放：完整执行轨迹可可视化浏览
- 支持人工纠正：任意步骤的 Tool Result 可被编辑，Agent 从该点继续推演

## 二、产品原理与主流模式的根本区别

### 2.1 控制流：谁在做决策？

| 维度 | 主流框架 | 本架构 |
|------|---------|--------|
| **决策中心** | 开发者写控制流，LLM 填充内容 | LLM 生成控制流，开发者只提供 Tool |
| **编排方式** | 显式图/链/角色定义 | 隐式，模型自主规划 Tool Call DAG |
| **子 Agent 调度** | 开发者配置 Handoff/路由规则 | 模型自主决定何时委派哪个 sub_agent |
| **用户交互时机** | 开发者在特定节点插入交互 | 模型自主决定何时 ask_user，参数决定策略 |
| **异常处理** | 开发者预设重试/降级逻辑 | 模型自主决策，日志支持事后纠正 |

**本质差异**：主流框架是"开发者编排，LLM 执行"；本架构是"LLM 编排，开发者提供能力单元"。

### 2.2 抽象层级：什么是基本单元？

| 框架 | 基本抽象 | 抽象层级 | 问题 |
|------|---------|---------|------|
| **LangChain** | Chain（链） | 高 | 链的组合复杂，调试困难 |
| **LangGraph** | Graph Node（图节点） | 高 | 图定义繁琐，LLM 能力被束缚在节点内 |
| **CrewAI** | Role（角色） | 高 | 角色固定，难以动态适应任务 |
| **AutoGen** | ConversableAgent（对话 Agent） | 中 | 对话驱动，非任务驱动 |
| **OpenAI Agents SDK** | Agent + Handoff | 中 | Handoff 显式定义，不够灵活 |
| **smolagents** | Code/Tool Agent | 中 | 仍有模式区分，不够统一 |
| **Phidata/Agno** | Tool + Agent | 中 | Agent 和 Tool 并存，未完全统一 |
| **本架构** | **Tool（唯一抽象）** | **低** | **极简，无层级，无模式** |

**关键洞察**：主流框架在 LLM 之上叠加了多层抽象，试图"帮助"开发者控制流程；本架构选择信任 LLM 的规划能力，只提供最基本的"能力单元"（Tool），让模型自主组合。

### 2.3 状态管理：执行过程如何被记录？

| 维度 | 主流框架 | 本架构 |
|------|---------|--------|
| **状态存储** | 内存为主，可选持久化 | 原生持久化，日志即状态 |
| **可恢复性** | 会话丢失 = 状态丢失 | 任意步骤可恢复，断点续传 |
| **可审计性** | 依赖外部日志系统 | 执行轨迹本身就是审计记录 |
| **可回放性** | 无原生支持 | 可视化回放，任意点编辑继续 |
| **调试方式** | 断点调试、日志分析 | 改 Tool Result 继续推演 |

### 2.4 人机协作：人类在系统中的位置？

| 维度 | 主流框架 | 本架构 |
|------|---------|--------|
| **协作模式** | 人下指令，机器执行（主从） | 人进入思维流，共同迭代（对等） |
| **交互时机** | 预设节点（审批/关卡） | 模型自主决定，参数化策略 |
| **纠正方式** | 外部干预，改代码重跑 | 原生能力，改数据继续 |
| **信任建立** | 预设规则限制（权限/模式） | 完全透明，随时可纠正 |
| **可视化** | 无，或仅最终输出 | 完整思维过程可视化 |

## 三、核心优势

### 3.1 极简架构

```
主流框架：用户 → 编排层 → Agent 层 → Tool 层 → 执行
本架构：  用户 → LLM → Tool → 执行
```

- **一种抽象**：Tool 是唯一的系统单元
- **一种循环**：Agent Loop（LLM 规划 → Tool 执行 → 结果反馈 → LLM 再规划）
- **一种存储**：Tool Call Log（日志即状态，状态即日志）

**优势**：学习成本低，扩展简单（新能力 = 新 Tool），维护负担小。

### 3.2 完全透明

- 每一步决策可见：为什么调用这个 Tool？输入是什么？输出是什么？
- 每一步可干预：发现错误？直接改 Tool Result，Agent 从该点继续
- 每一步可审计：完整轨迹满足企业合规要求

**优势**：信任不是建立在"限制"上，而是建立在"可见"上。

### 3.3 弹性恢复

- 异常不致命：任意步骤失败，从断点重试，无需全量重启
- 状态不丢失：持久化日志保证任务不中断
- 人工可接管：系统卡住？人类纠正一步，Agent 继续

**优势**：生产环境鲁棒性高，适合长周期复杂任务。

### 3.4 动态适应

- 子 Agent 委派：模型自主决定"这个任务需要哪个专家"
- 用户交互策略：模型自主决定"现在需要问用户什么"
- 资源动态分配：模型自主决定"这个任务需要多少算力"

**优势**：无需预设规则，系统自适应任务复杂度。

### 3.5 知识沉淀

- 成功的工作流 = 可复用的 Tool Call 序列
- 可保存为模板，分享为最佳实践
- 可转化为自动化脚本，无需人工介入

**优势**：Agent 不是一次性工具，是组织知识的积累载体。

## 四、核心组件设计

### 4.1 Tool 统一接口

```python
class Tool:
    name: str                    # 工具标识
    description: str             # 模型可读的功能描述
    parameters: JSONSchema       # 参数 schema，模型生成调用
    permissions: PermissionSet   # 权限元数据（读/写/执行）
    resource_quota: ResourceQuota # 资源配额（超时、内存、并发）

    def execute(self, params: dict, context: ExecutionContext) -> ToolResult:
        # 执行逻辑
        pass
```

### 4.2 关键 Tool 设计

#### `sub_agent` Tool

```python
{
    "name": "sub_agent",
    "description": "委派任务给专门的子 Agent",
    "parameters": {
        "agent_id": "string",           # 子 Agent 标识
        "task": "string",               # 任务描述
        "context": "object",            # 上下文传递（唯一输入）
        "resource_quota": {
            "max_steps": 50,            # 最大执行步数
            "timeout_seconds": 1800,    # 超时时间
            "memory_limit": "2GB"       # 内存限制
        },
        "permissions": ["read", "write"],  # 权限范围
        "checkpoint_interval": 5,       # 检查点间隔（步）
        "on_timeout": "persist_and_notify"  # 超时策略
    }
}
```

#### `ask_user` Tool

```python
{
    "name": "ask_user",
    "description": "向用户请求信息或确认",
    "parameters": {
        "question": "string",           # 问题描述
        "options": [                    # 选项列表
            {"id": "A", "label": "选项A", "is_default": false}
        ],
        "recommendation": "string",     # 系统推荐（有推荐即默认继续）
        "fields": [                     # 输入字段
            {"name": "detail", "required": false, "default": ""}
        ],
        "on_no_response": "block" | "use_default" | "escalate"
    }
}
```

### 4.3 调度器设计

```python
class Scheduler:
    def execute(self, dag: ToolGraph) -> ExecutionResult:
        for step in dag.topological_sort():
            # 检查点：持久化当前状态
            if step.is_checkpoint:
                self.checkpoint_store.save(step)

            # 执行 Tool Call
            result = step.tool.execute(step.params, step.context)

            # 日志记录
            self.execution_log.append({
                "step_id": step.id,
                "tool": step.tool.name,
                "input": step.params,
                "output": result,
                "timestamp": now(),
                "latency_ms": result.latency
            })

            # 异常处理
            if result.is_error:
                if step.retry_policy.can_retry:
                    step.retry()
                else:
                    raise ExecutionError(step, result)

        return ExecutionResult(self.execution_log)
```

### 4.4 可视化回放系统

```
无限画布
├── 时间轴（Tool Call 序列）
│   ├── 每一步展开：输入 / 输出 / 模型决策 / 耗时
│   ├── 状态快照：任意点可恢复
│   └── 分支探索：改 A 走左边，改 B 走右边
├── 实时协作
│   ├── 多人同时查看同一 Agent 思维流
│   ├── 各自纠正不同节点
│   └── 冲突检测与分支合并
└── 导出
    ├── 完整审计轨迹（合规）
    ├── 可复现脚本（自动化）
    └── 知识沉淀（工作流模板）
```

### 5.3 为什么主流框架没有走这条路

| 原因 | 说明 |
|------|------|
| **历史惯性** | 从 Chatbot 渐进演化，天然带"对话轮次"基因 |
| **产品定位** | 主流是"辅助工具"，非"自主执行器" |
| **控制欲** | 开发者/框架设计者想要显式控制流 |
| **可解释性** | 企业需要"为什么这样做"的答案，传统框架用"模式"间接提供 |
| **基础设施门槛** | 完整可视化+可回放需要极强的工程投入 |
| **生态壁垒** | 现有生态（插件、文档、社区）围绕模式构建 |

## 六、Code/Work 模式说明

> **Code 和 Work 不是两种模式，而是同一架构在不同参数配置下的自然表现。**

| 场景 | 参数配置 | 表现特征 |
|------|---------|---------|
| **Code**（实时辅助） | 浅 DAG、硬超时、无默认推荐、必填字段 | 即时响应，高交互频率 |
| **Work**（后台任务） | 深 DAG、软超时、有默认推荐、全可选字段 | 异步执行，低交互频率 |

**关键**：无需显式 `mode` 参数，系统根据上下文自动调整。`ask_user` 的 `recommendation` 字段存在与否即信号——有推荐是 Work 语义（默认继续），无推荐是 Code 语义（必须阻塞）。

## 八、总结

### 核心观点

> **在"一切皆工具，模型驱动一切，日志原生持久化"的架构下，Code 和 Work 是伪区分。**
>
> 同一套架构，同一套 Tool，不同参数配置，自然呈现不同行为特征。
>
> 系统不需要显式模式，只需要：统一的 Tool 抽象、参数化的调度策略、完全透明的执行轨迹。

### 架构优势

1. **极简**：一种抽象（Tool），一种循环（Agent Loop），一种存储（日志）
2. **透明**：每一步可观察、可编辑、可延续
3. **弹性**：异常可恢复，错误可纠正，状态不丢失
4. **动态**：模型自主规划，自适应任务复杂度
5. **沉淀**：工作流即知识，可复用分享

### 与主流的根本差异

| | 主流框架 | 本架构 |
|--|---------|--------|
| **哲学** | 把 LLM 当作组件，开发者编排控制流 | LLM 是唯一的决策中心，控制流由模型生成 |
| **抽象** | Agent/Chain/Graph/Role 多层抽象 | 单一 Tool 抽象 |
| **人机关系** | 人下指令，机器执行 | 人进入机器思维流，共同迭代 |
| **调试** | 外部干预，改代码重跑 | 原生能力，改数据继续 |
| **信任** | 预设规则限制 | 完全透明，随时纠正 |
